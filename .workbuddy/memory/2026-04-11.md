## 修复3：前端流水线 pipeline.js 接口调用去重优化

### 审计发现
进入流水线页面时，存在大量冗余/重复接口调用：
1. **`getResult()` 无效探针**：在 `loadDocumentInfo()`、`loadChunks()`、`loadGenerationResults()` 各调用一次，新文档必返回400错误，白白浪费请求
2. **`nextStep()` 与 `loadXxx()` 双重调用**：点击"下一步"时 `nextStep()` 调一次API，然后 `renderStepContent()` 的 `loadXxx()` 又调一次相同的API
3. **步骤4 每次渲染都调 import**：已完成入库的文档每次进入页面都重新调 `/import/{id}`

### 修复内容
| 方法 | 修改 | 节省 |
|------|------|------|
| `loadDocumentInfo()` | 去掉 `getResult` 探针，直接用 `getPreview` | -1次请求 |
| `nextStep()` | 去掉所有内部API调用，只做状态推进+导航 | 步骤2/3各-1次重复调用 |
| `loadChunks()` | 去掉 `getResult` 探针，直接调 `split` | -1次无效探针 |
| `loadGenerationResults()` | 去掉 `getResult` 探针，直接调 `generate` | -1次无效探针 |
| `loadImportResults()` | 已完成文档直接展示缓存数据，不调import | -1次重复导入 |

### 优化前后对比（新上传文档走完全流程）
- **优化前**：~14次请求（含6-7次浪费）
- **优化后**：~7次请求（全部有效）

## 修复6：已完成文档重复入库（状态降级 bug）

### 问题现象
文档已 `completed`，再次进入流水线页面时完整重走了一遍嵌入生成+Milvus导入（7秒+），导致 Milvus 中出现大量重复向量。

### 根因分析（关键 bug）
`generate` 接口第198行的逻辑：
```python
if doc["status"] != "generated":
    db.update_document(file_id, status="generated")
```
当文档状态为 `completed` 时，`"completed" != "generated"` 为 True → **把状态从 completed 降级回 generated！**

攻击链路：
1. 文档已 `completed`
2. 前端调 `generate` 拿缓存数据 → 后端返回缓存 ✅ 但同时把 status 降级成 `generated` ❌❌❌
3. 前端调 `import` → 后端看到 `status='generated'` 而非 `'completed'` → 完整重跑嵌入+导入

### 修复内容

#### 1. backend/api/processing.py — generate 接口（根因修复）
```python
# 旧：会把 completed 降级为 generated
if doc["status"] != "generated":  # completed != generated → True!

# 新：保护 completed 状态不被降级
if doc["status"] not in ["generated", "completed"]:
```

#### 2. backend/api/processing.py — import 接口（服务端硬保险）
在函数入口最顶部新增 `completed` 状态的即时拦截，不依赖前端传值：
- 如果 `doc["status"] == "completed"` → 直接从 DB 统计数据返回结果，跳过所有嵌入和导入逻辑
- 即使前端因为任何原因误发请求，后端也能正确幂等处理

### 教训
**状态机只能向前推进，绝不能向后降级！** 生成接口只应负责推进到 generated，不应触碰已处于更高状态的文档。

## 修复7：Step 3（生成增强）Chunk 列表预览格式错误

### 问题
Step 3 左侧 Chunk 选择器中，每个 chunk 显示的是嵌套缩进的原始内容或仅"文本"两字，而 Step 2 的 Chunk 列表是正常的一行内容预览。

## 修复8：Markdown表格导致Chunk预览显示异常（嵌套/合并）

### 问题
文档包含Markdown表格时，Step 2和Step 3的chunk列表预览出现异常，表现为：
- 表格内容"合在一起"，多个chunk看起来像是嵌套
- 表格分隔符（`|`）和换行导致预览显示混乱
- `-webkit-line-clamp: 2` CSS属性使表格行显示为两行，造成视觉上的嵌套感

### 根本原因
1. **`_stripMarkdown()` 未处理表格**：原方法只处理标题、粗体、列表等，忽略了表格标记
2. **表格行合并**：方法用 `join(' ')` 合并所有行，表格的多行内容被合并成长字符串
3. **预览截断逻辑**：表格内容通常较长且结构化，简单截断可能保留不完整的表格标记

### 修复方案
增强 `_stripMarkdown()` 方法，专门处理Markdown表格：

#### 表格处理逻辑
1. **识别并跳过表格分隔行**：匹配 `|---|` 或 `|=|=|` 等模式的行直接忽略
2. **提取单元格内容**：对于表格内容行，移除 `|` 分隔符，分割单元格，取前两个单元格内容
3. **压缩表格内容**：将多个单元格内容合并为 `"单元格1 - 单元格2"` 格式，避免过长
4. **总体长度限制**：最终结果超过200字符时截断，防止表格内容占据整个预览

#### 关键代码改动
```javascript
if (cleaned.includes('|')) {
  // 移除表格分隔行
  if (cleaned.match(/^[\s\|]*[-=]+[\s\|]*$/)) {
    return ''; // 表格分隔行直接忽略
  }
  // 提取单元格内容
  cleaned = cleaned.replace(/^\||\|$/g, '');
  const cells = cleaned.split('|').map(cell => cell.trim()).filter(cell => cell);
  if (cells.length > 0) {
    // 取前两个单元格内容，避免表格内容过长
    cleaned = cells.slice(0, 2).join(' - ');
  }
}
```

### 预期效果
- 表格内容在预览中显示为简洁的文本摘要
- 避免表格标记和结构导致视觉混乱
- 保持预览的可读性和一致性

## 修复9：Chunk 预览嵌套问题（HTML 转义缺失）

### 问题
Step 2 和 Step 3 的 chunk 列表中，某些 chunk（如 #4、#5）被错误地嵌套在前一个 chunk 内部。

### 根本原因
chunk 内容中可能包含 HTML 标签字符（如 `<table>`、`<div>`、`<tr>` 等），直接插入 `innerHTML` 时，浏览器会将其解析为真实 HTML 元素，破坏 chunk-item 的 DOM 结构边界，导致视觉上的"嵌套"。

### 修复方案
1. **新增 `_escapeHtml()` 方法**：使用 DOM API 将文本中的 `<`、`>`、`&`、`"`、`'` 等特殊字符转义为 HTML 实体
2. **修改 `updateChunkList()`**：预览文本和类型标签都先经过 `_escapeHtml()` 转义
3. **修改 `updateGenChunkList()`**：同上，确保 Step 2 和 Step 3 的渲染逻辑一致

### 关键代码
```javascript
_escapeHtml(text) {
  const div = document.createElement('div');
  div.textContent = text;
  return div.innerHTML;
}

// 渲染时：
const cleanPreview = this._escapeHtml(this._stripMarkdown(rawPreview).substring(0, 80));
```

### 教训
使用 `innerHTML` 插入用户/文档内容时，**必须先转义 HTML 特殊字符**，防止 XSS 攻击和 DOM 结构破坏。即使是内部内容，也要做防御性处理。
